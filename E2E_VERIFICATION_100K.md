## Issues Found

### Issue 1: Docker RAM Budget Tight for 5 Fat Services (ADVISORY)

**Severity:** Low (advisory only)

5 services x 768MB = 3840MB + infrastructure (512+256+256) = 4864MB total. This exceeds the TECH-006 budget of 4.5GB by 364MB. In practice, Docker `mem_limit` is a hard cap but actual resident memory is typically 30-60% of the limit. For 5 bounded-context services, actual memory usage will likely be under 3GB. The risk is low but worth noting.

**Mitigation:** If memory becomes an issue during integration tests, the 768MB limit can be reduced to 640m for a 5-service config, or infrastructure containers can be trimmed (postgres 384m instead of 512m).

### Issue 2: `_DOTNET_DOCKERFILE_TEMPLATE` Uses Generic DLL Name (ADVISORY)

**Severity:** Low

pipeline.py:374 `ENTRYPOINT ["dotnet", "App.dll"]` uses a hardcoded `App.dll` name. The actual DLL name depends on the .csproj project name. The compose_generator.py:529 version has the same issue. Builders will likely create their own Dockerfile with the correct DLL name, so this only affects fallback generation.

### Issue 3: No `_STACK_DEFAULT_PORTS` Entry for Dotnet

**Severity:** Low

compose_generator.py:31-35 `_STACK_DEFAULT_PORTS` does not have a "dotnet" entry. The `_get_internal_port()` method at line 297 falls back to 8080 for unknown stacks, which IS the correct port for ASP.NET Core. So functionality is correct but the fallback path is being used rather than an explicit mapping.

### Issue 4: `build_cross_service_standards()` Has No Stack Parameter

**Severity:** Low (informational)

cross_service_standards.py:901 `build_cross_service_standards(service_id, is_frontend=False)` does not take a `stack` parameter. All services receive all standards (Python + TypeScript + C# examples). This means Python services see C# examples and vice versa. This is harmless -- extra examples don't cause errors -- but adds ~1.5KB of irrelevant content to each CLAUDE.md.
