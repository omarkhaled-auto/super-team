### A5 Detail: Extra Entity "Vendor"

The PRD text contains: `"Client | A customer or vendor entity"`. The prose pattern extractor (Strategy 3, Pattern C) matches `"Vendor entity"` and creates a spurious 6th entity. This is a known limitation of the regex-based prose entity extraction.

**Impact:** LOW. The extra entity is assigned to the Miscellaneous boundary and does not interfere with the core 5 entities. Existing tests in `test_prerun_fixes.py` do not assert an exact entity count of 5.

### A8 Detail: Duplicate State Machines

The parser produces 6 state machine entries because two strategies independently extract the same machines:
- Strategy 4 (heading format): Extracts "InvoiceStatus", "JournalEntryStatus", "FiscalPeriodStatus"
- Strategy 5 (transition sections): Extracts "Invoice", "JournalEntry", "FiscalPeriod"

The domain modeler correctly deduplicates by matching on the entity name, so each DomainEntity gets exactly one StateMachine. The Strategy 5 versions are preferred because they have meaningful trigger labels.

**Impact:** NONE. The domain model correctly has 3 state machines with all expected states and transitions.

### A11 Detail: No AsyncAPI Stubs

The `generate_contract_stubs()` function in `src/architect/services/contract_generator.py` only produces OpenAPI specs. It does not generate AsyncAPI specs even when the PRD mentions a message broker (RabbitMQ) and the domain model has event-driven entities.

**Impact:** MEDIUM. AsyncAPI contracts would need to be generated by a separate mechanism or added to the contract generator. For the LedgerPro PRD, the MCP server's `decompose` endpoint would need enhancement to produce AsyncAPI stubs alongside OpenAPI stubs.

## Remaining Risks for Full Pipeline Run

### HIGH Risk

1. **Entity assignment imbalance:** Ledger Service and Period Service are generated with 0 entities. The PRD defines "Ledger Service" for journal entries and "Period Service" for fiscal periods, but the boundary algorithm assigns JournalEntry to Invoice Service (relationship-based proximity) and FiscalPeriod to Miscellaneous (no explicit bounded context assignment). This means the builder agent for ledger-service and period-service would generate empty services with only health endpoints.

2. **Duplicate state machines (6 instead of 3):** While the domain modeler deduplicates correctly, other pipeline stages that read `parsed.state_machines` directly may process 6 entries instead of 3. The FiscalPeriodStatus variant from Strategy 4 is also missing the `closing -> open` reverse transition (3 transitions vs. 4 from Strategy 5).

### MEDIUM Risk

3. **No AsyncAPI contract generation:** The contract generator does not produce AsyncAPI stubs. If the pipeline run expects AsyncAPI contracts for event-driven communication (RabbitMQ), this could cause downstream failures in the contract registration phase.

4. **Spurious "Vendor" entity:** The extra entity could confuse the builder agent, creating API endpoints and database tables for a non-existent domain concept. The entity has no fields and no relationships.

### LOW Risk

5. **Empty entity fields:** All 5 legitimate entities have 0 fields because the LedgerPro PRD defines entities only via a summary table (Entity | Description) without field-level tables. The builder agent would need to infer or generate fields.

6. **No frontend service detected:** The LedgerPro PRD does not specify a frontend. The pipeline would create only backend services.
